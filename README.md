# 设计稿自查（design-selfcheck）

用于在设计交付前，对 B 端电商供应链产品的设计稿做结构化自查。这个 skill 会先收集需求背景，再结合设计稿内容和内置清单完成分析，最后输出可直接执行的分级问题与优化建议。

## 核心能力

- 基于 Figma 链接、页面截图或设计描述，理解当前设计方案。
- 在分析前主动收集需求背景，避免脱离业务语境机械审查。
- 按需求、框架、交互、界面、交付与 B 端补充维度做系统性检查。
- 输出面向设计师的评审报告，问题按严重缺陷、一般缺陷、优化建议分级。
- 在设计稿无法直接读取时，允许基于截图或描述推断，并明确标注推断前提。

## 适用场景

- 设计方案完成后，准备交付研发前做一次系统性自查。
- 用户提供 Figma 链接，并希望 AI 帮忙 review 设计稿。
- 用户同时给出 PRD 和设计稿，希望结合需求背景审查方案质量。
- 需要从 B 端业务效率、信息密度、异常状态、批量操作等视角检查方案。

## 不适用场景

- 用户只想理解 PRD、拆解页面范围或评估设计工作量，此时更适合 `prd-design-brief`。
- 用户重点关注设计稿中的 mock data 是否真实一致，此时更适合 `figma-mockdata-reviewer`。
- 用户没有设计稿、截图或可描述的页面内容，无法开展有效审查。

## 输入与前置条件

- 设计稿输入优先级：`Figma 链接` > `页面截图` > `Figma 导出 JSON` > `用户文字描述`。
- 背景信息优先级：`用户补充说明` > `PRD / 需求文档`。
- 如果设计稿无法直接访问，应先请求截图或补充描述，再继续分析。
- 执行阶段 3 时，skill 会读取 `reference/checklist/` 下的自查清单作为内部分析框架。

## 使用方式

### 推荐指令

```text
帮我自查这个设计稿：https://www.figma.com/design/...
```

```text
review 一下我的设计，这次需求主要面向采购专员，目标是提升批量处理效率。
```

```text
这是 PRD 和设计稿链接，请按设计自查的方式帮我做一份评审报告。
```

### 建议同时提供的信息

- 需求背景：这次方案要解决什么业务问题。
- 目标用户：主要角色、使用场景和操作频率。
- 设计目标：本次最关注的体验目标或业务指标。
- 功能范围：涉及的核心模块、关键流程和重点页面。
- 约束条件：技术限制、品牌规范、上线时间或已有设计系统约束。

## 工作流程

1. 识别并读取设计稿内容；若无法直接访问，则请求截图或补充描述。
2. 先收集需求背景，必要时从 PRD 中提取并整理关键信息。
3. 读取 `reference/checklist/README.md` 及对应检查项，作为内部分析依据。
4. 结合业务背景、目标用户与 B 端场景，对设计方案逐项审查。
5. 输出结构化评审报告，并给出按优先级排序的改进建议。

## 输出结果

- `需求背景摘要`：产品目标、核心用户、关键约束。
- `设计推导`：用户目标、业务目标、设计目标、设计策略、方案概述。
- `设计缺陷分析`：聚焦用户卡点与技术可行性挑战。
- `其他设计方向参考`：提供 2-4 个替代思路。
- `综合建议`：按“紧急重要 / 重要不紧急 / 次要”给出可直接执行的动作。

问题分级规则如下：

- `严重缺陷`：影响核心流程或用户认知，必须修改。
- `一般缺陷`：存在不一致或体验欠佳，建议修改。
- `优化建议`：可提升体验，但不影响当前主流程。

## 目录说明

- `SKILL.md`：skill 的触发条件、流程和输出规范。
- `README.md`：当前这份面向使用者的能力与用法说明。
- `reference/checklist/`：自查知识库与分维度检查清单。
- `使用指南.md`：更偏新手安装和上手流程的长文说明。

## 注意事项

- 不能跳过背景收集直接出报告，否则结论容易失真。
- 自查清单只作为内部分析工具，不应在最终报告中暴露条目编号。
- 如果结论基于截图或文字描述推断，报告中必须明确说明。
- 输出目标是“让设计师读完就能改”，因此问题、影响和建议都应写具体。
